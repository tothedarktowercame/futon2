## 1. IDENTIFY (shape-first)

### Motivation — the discrepancy

When `M-war-machine-first-outing` cleared the open-action queue, the only remaining recommendation was `:learn-action-class`. The WM was **starved of input sources, not done** (`project_war_machine_first_outing`; quantified in the expectations doc: 100% bootstrapping, 0% outward, queue exhausted into `:learn-action-class` before any `:open-mission` surfaced). An operator reasonably expects an active-inference agent to have "a built-in way to learn things." It does — but only *within* its model:

- **Within-model learning is present and honest:** belief update (R1/R3), structure/parameter learning via reliability posteriors (R7 adaptive precision; R12 dual-loop over inner hyperparameters). These pick among, and re-weight, the policies in the model's existing support.
- **What is absent:** any endogenous way to enlarge the *support itself* — to add a new typed-edge family / action class. Today that is an **external handoff**: `:learn-action-class` is a doorbell the operator answers. `niche-construction.flexiarg` states this as a scoped property of the current architecture (substrate authored by other parties), **not** a theorem about AIF — and the flexiarg was corrected 2026-05-31 to make that scope explicit.

The discrepancy: AIF theory has a built-in story for the meta-level (hierarchical priors that re-weight uncertainty-tolerance — Kiverstein & Miller; variational niche construction that redraws the blanket to include the constructed niche — Constant/Ramstead/Veissière/Friston). **Our implementation does not yet use it.** The gap is in the wiring, not the framework.

### The central distinction: backlog vs. installed priorities

Two ways to "feed the starved WM," and they are not the same mission:

- **(A) Big programmatic backlog.** Wire each of the five frontier sources as an enumerator-proposer (in the style of v0.18 `mission-registry` / v0.19 `pattern-registry`). The candidate set grows; the WM has more concrete `:open-mission` / `:fire-pattern` / `:address-sorry` targets to rank. This is **within-model** — R6 action-*proposal* scaling over a *fixed* set of action-classes. It produces a longer to-do list. It does **not** extend the support and it does **not** teach the WM what to value.

- **(B) Learn and install new priorities.** The WM *learns what to value* (intrinsic-value / preference-prior learning; a playful exploration prior) and can *install a new priority* — a new action-class, or a re-weighting that changes which kinds of work it seeks. This is the **boundary / meta-level** move: the support and the value-weighting become things the agent infers, not constants the operator sets.

**Joe (2026-05-31): (B) is the interesting target.** (A) is useful plumbing and may be a prerequisite, but it is not what `aif2` is *for*. The five sources below are therefore treated as **idea-stock for (B)** — raw material from which new priorities might be learned — not as a backlog whose completion would count as success. This is tension T4, and it is the mission's load-bearing commitment.

### Q1 (= T1) — More R-criteria, or a new site? **Answer: a new site, plus one R12 extension. CONFIRMED.**

Every existing R-criterion operates *over a fixed support*. There is no R-criterion whose remit is "may the support itself change?" — that question lives at AIF+ **I1 (boundary integrity)** and **I3 (timescale)**, both unmechanised (`README-aif+.md` §3). Concretely:
- `inventory-as-hidden-state` is a natural **extension of R12** (R12's slow-loop apparatus already exists; it just infers the wrong quantity — see Q2). Candidate **R13**.
- `boundary-redraw-admissibility` is a **genuinely new tier** — not a 13th item in the R-list, because the R-list presupposes the boundary it would govern. It is the I1 mechanisation.

So the IDENTIFY hypothesis holds: **one R12-extension + one new boundary-tier site.** Not "more R's."

### Q2 — Where does the action-class inventory live, and what would make it hidden state?

**The inventory is CODE, not data — and it is enumerated in THREE places, not one:**
1. `forward-model/action-types` — a `def` literal set `#{:no-op :address-sorry :open-mission :fire-pattern :learn-action-class}` (`forward_model.clj:25-30`). The support proper.
2. `defmulti` dispatch tables keyed by type: `predict-effects` (`forward_model.clj:52-103`), `can-propose?` (`105-118`), `can-execute?` (`128-142`).
3. The slow loop **re-hardcodes** the class list: `classes [:address-sorry :open-mission :fire-pattern]` (`wm_outer_loop.clj:289`) + `substrate-dispatch` (`240-243`).

Extending the support today = editing ≥3 namespaces + reloading the JVM. The WM cannot `conj` to it. This is exactly the flexiarg's "operations on values within the niche but not on the meta-structure of the niche," made concrete.

**What would make it hidden state:** collapse those three code sites into a single **data-driven action-class registry** (a map `class → {:predict-effects-spec, :can-propose-pred, :intrinsic-value-posterior, :status}`) that the slow loop can both *read* and *propose mutations to*, guarded by a `boundary-redraw-admissibility` gate, with candidate classes seeded from the five idea-sources. `predict-effects` can stay a `defmulti` (the registry holds a *reference*, not the impl) — the change is bounded.

**Coupling is small** — `action-types` is referenced directly in only ~2 source files (`forward_model.clj`, `action_proposer.clj:39`) + 1 test (grep verified). This bears on **T2**: an *in-place* extension is feasible; a separate `aif2` library is a choice about organisation, not a necessity forced by coupling.

### The R12 apparatus is a near-perfect template — one level too low (the key finding)

`intrinsic-values` already treats **per-class value** as hidden state: Beta(α,β) posteriors keyed by class (`intrinsic_values.clj:59-74`), persisted as `code/v05/wm-hyperparameter-update` hyperedges in futon1a XTDB, bootstrap-replayed on JVM start (`rehydrate-from-store!`, `223-228`), updated daily by the outer loop. `credit-for` defaults a *new* class to Beta(1,1)=0.5 (`68-74`). The slow clock (`wm_outer_loop.clj`) reads the trace window, extracts emissions per class (`extract-emissions`, `120-156`), derives operator follow-through from git (Stream B — `count-sorry-closures-in-window`, `count-mission-additions-in-window`), updates Beta, persists.

**This is the exact apparatus `inventory-as-hidden-state` needs — but it infers value-OF-class, not existence-OF-class.** The class set is a *precondition* of the Beta table, not an inferred quantity. Porting it *up one level* — a posterior over *candidate classes* (propose → prior → accumulate model-evidence/follow-through → install or prune) — is the cheapest path to R13, and reuses the Beta + XTDB + outer-loop + Stream-B template almost directly.

### Surprise 1 — the boundary is invisible to the agent

`action_proposer/gap-actions` (`action_proposer.clj:34-46`) surfaces `:learn-action-class` only for `(disj fm/action-types :no-op :learn-action-class)` filtered to where `can-propose?` is false. So the WM can recommend "enable the *declared* class `:fire-pattern`," but can **never** surface "you lack an action-type for X." The unknown-unknown — a genuinely missing class — is structurally invisible. Even the self-model of capability gaps is bounded by the fixed support. **Consequence for T4:** today's `:learn-action-class` is an (A)-flavoured signal (enable a declared class), *not* (B) (install a new kind). The five idea-sources are what would feed candidate-class generation — the missing (B) input.

### Surprise 2 — preferences are static data (a lighter (B) locus)

`preferences`, `avoided-states`, `mode-prior`, `pragmatic-weights`, `channel-health-signs` are `def` literals sourced from `war-machine-terminal-vocabulary.edn` (`preferences.clj:8-77`). R12 does *not* infer these (contract §R12 confirms "static per-channel hyperparameters remain a separate open direction"). ## 3. DERIVE — Part 1: the design spine

*Authored 2026-05-31 (claude-1 + Joe). DERIVE opens with the unifying architectural decision; remaining DERIVE checklist items (full entity/relation types, invariant formalisation, futon5 AIF+ wiring diagram, GF capability-preservation matrix, PSRs) are subsequent cars — see "Remaining DERIVE cars" below. ### The unifying move: one primitive, three strata (+ one optional)

Both gaps — and Joe's **meta-meta-learning** extensibility constraint — collapse to a single observation: *the WM is a stack of "what can be proposed," and each layer is currently a different mechanism.* Make them **one self-similar mechanism**:

| Stratum | "what can be proposed" | today | disposition |
|---|---|---|---|
| **S3 Targets** | which sorry/mission/pattern to act on | **data** (proposers enumerate from substrate) | already extensible — the model |
| **S2 Action-classes** | which *kinds* of action exist | **code** (`forward-model/action-types` def + `defmulti` arms) | **Gap-2**: make it an `extensible-registry` (hidden state) |
| **S1 Proposers** | which *sources of candidates* run | **code** (literal `[bootstrap pattern mission sorry]` at `war_machine.clj:3296`) | **Gap-1**: add a **tension-proposer**; make the proposer-set an `extensible-registry` |
| **S0 Code** (optional) | the *implementations* themselves, over a metric'd code-manifold | code | **optional extra** — integrate M-differentiable-code (kept optional: that mission is early-pilot) |

### The primitive: `extensible-registry`

One entity type, instantiated once per stratum. A registry is a map `id → entry`; an entry is:

```
{:id            <keyword>
 :producer      <a reference: how this entry yields its outputs —
                 for S3 an enumerator, S2 a predict-effects/can-propose spec,
                 S1 a propose-fn; held by reference so the impl can stay defmulti>
 :credit        <Beta(α,β) posterior — the R12 intrinsic-values apparatus ported up>
 :admissibility <predicate: may this entry act / be installed? — the consent-gate locus>
 :status        <:active | :candidate | :pruned>}
```

- **`:credit`** is the existing R12 machinery (`futon2.aif.intrinsic-values`: Beta posteriors, XTDB-persisted as `code/v05/wm-hyperparameter-update`, bootstrap-replayed, daily outer-loop) **generalised from "per action-class" to "per registry entry at any stratum."**
- **`:admissibility`** is the `boundary-redraw-admissibility` criterion from IDENTIFY — the I1-tier gate. Installing/activating an entry (especially a `:candidate`) is a boundary redraw; consent-gated, with supervised→autonomous migration ([[project_consent_gate]]).

### How each gap is one instance

- **Gap-1 — tension-proposer (first new S1 entry).** A proposer whose `:producer` reads ∇T over the manifold (generalising `mission_delta_t` beyond sorry-related missions) and **emits candidate actions at high-tension regions**. This is the dish-on-the-menu fix: tension now *generates*, not just *ranks*. Works with **today's** S2 inventory (can emit `:open-mission`/`:address-sorry`/`:fire-pattern` at steep-∇T regions) — which is why Gap-1 is independently valuable.
- **Gap-2 — candidate-class posterior (S2 as registry).** `action-types` becomes the S2 registry; new action-classes enter as `:candidate` entries, accrue `:credit` via model-evidence/operator-follow-through (the R12 template, ported *up one level* from value-of-class to existence-of-class), and are installed when admissibility passes. Unifies the three enumeration sites MAP found (`action-types`, `gap-actions`' iteration, the outer-loop's hardcoded `classes`/`substrate-dispatch`) into one data source.

### Meta-meta-learning, made concrete (and why it terminates)

Joe's constraint: the solution must be extensible "via meta-meta-learning." Concretely: **S1 and S2 use the identical `extensible-registry` primitive**, so —
- *learn-to-value-targets* (≈ R12 today),
- *learn-to-install-action-classes* (Gap-2), and
- *learn-to-install-proposers* (the S1 registry being extensible)

— are **one mechanism applied at three heights.** A proposer that proposes *new proposers* is itself just a registered S1 entry with a `:credit` and an `:admissibility`. 

**Why this doesn't merely push the false-floor up a rung (the key argument):** the floor-raising operation is the *same operation* at every stratum, governed by the *same* admissibility gate and the *same* credit-learning loop. The recursion lives in the **data** (registry entries), not in new **code** per level. So "add a stratum" or "add an entry" is always the same finite move — there is no n-th-level mechanism that needs hand-coding, which is exactly what would re-introduce a false floor. *(This is the `feedback_admissibility_as_recursion_safety` generalisation: per-entry admissibility IS the cycle-stopper for self-modifying actions — now lifted to per-stratum.)*

### Central design decision (IF/HOWEVER/THEN/BECAUSE)

**IF** we made Gap-1 and Gap-2 two bespoke features (a tension-proposer hardcoded into the proposer vector; a separate action-class-learning loop), **HOWEVER** that would leave the proposer-set and the extension-machinery themselves fixed — the false floor would reappear at the next level up, violating Joe's meta-meta-learning constraint, **THEN** we instead build **one `extensible-registry` primitive and instantiate it at S1 and S2 (S3 already is one)**, **BECAUSE** uniformity across strata is precisely what makes the system extensible "all the way up" with finite code, and it lets us reuse the existing R12 credit apparatus and the consent-gate admissibility at every level rather than re-deriving them per gap.

### T2 resolution (proposed)

`aif2` is a **new namespace family** `futon2.aif2.*` (the `extensible-registry` primitive + the per-stratum instances + the tension-proposer), **reusing** `futon2.aif.*` unchanged as the reduced/reference model — `aif`'s `defmulti` machinery stays as the `:producer` implementations that registry entries *reference*. Bayesian model-reduction relationship preserved: `aif` is `aif2` with all registries frozen to their current literal contents. *In-place* (not a separate repo/lib). This satisfies the subsumption discipline: both `aif` (frozen) and `aif2` (extensible) stay live.

### Checkable invariants (preview — full formalisation is the next car)

- **INV-provenance:** every candidate carries `{proposer-id, action-class, target}` — no orphan candidates.
- **INV-no-bypass:** a tension-*generated* candidate still passes admissibility + EFE ranking; tension never short-circuits the gate.
- **INV-uniformity (the meta-meta guarantee):** S1 and S2 share the registry protocol — adding an entry at either uses the same `register!`/`:admissibility`/`:credit` path (testable: no stratum-specific install code).
- **INV-reduction (GF):** with all registries frozen to current literal contents, `judge` reproduces today's behaviour exactly (the basic `aif` model is preserved — capability-preservation tripwire).
- **INV-consent:** installing/activating a `:candidate` entry is consent-gated; autopen-eligibility rule is per-stratum admissibility (T3).

## 3c. DERIVE — Part 3: registry entity/relation types + the ∇T reader (car 1)

### Prior-art reconciliation (I-4 — three library patterns read before defining; the primitive *composes* them)

- **`exotic/full-lift-registry`** → entries use **homoiconic, inspectable expressions** for the producer (code-as-data, not opaque fns) + per-entry `:provenance {:evidence :version :source}`. **This homoiconicity is load-bearing**: it *is* the "recursion-in-data" property the meta-meta argument needs — entries are manipulable by the same system that runs them.
- **`aif/candidate-pattern-action-space`** → action selection = a **bounded candidate set with explicit inclusion/exclusion reasons** + coverage tracking (entered-vs-chosen). Candidates carry inclusion provenance (→ INV-provenance).
- **`aif/evidence-precision-registry`** → trust as an explicit, tunable **precision** surface. A *distinct axis* from credit: the tension-proposer has TWO trust quantities — `:credit` (is this entry worth running? Beta, from `intrinsic-values`) and `:precision` (how reliable is its signal?).

### Entity types

**`registry-entry`** — the primitive; one shape, all strata:
```
{:id            <namespaced kw: s1/… | s2/…>     ; shape-first stratum namespace
 :stratum       <:s1-proposer | :s2-action-class>
 :producer      <homoiconic expr / ref — S1: a propose-form; S2: a
                 predict-effects + can-propose ref (defmulti stays, held by ref)>
 :credit        <Beta(α,β) — futon2.aif.intrinsic-values, keyed by :id (not class)>
 :precision     <optional Π — signal reliability (evidence-precision-registry kin)>
 :admissibility <pred / consent-policy ref — the boundary-redraw gate>
 :status        <:active | :candidate | :pruned>
 :provenance    {:evidence […] :version <int> :source <str> :inclusion-reason <str>}}
```
- **`candidate-action`** (what an S1 producer emits): `{:type <s2 entry id> :target <entity> :weight :provenance {:proposer-id :inclusion-reason} :structural-pressure-per-action <n>}`.
- **`candidate-entry`**: a `registry-entry` with `:status :candidate` — what `C-candidate-gen` emits, awaiting `C-admissibility`.
- **`tension-reading`** (∇T-reader output): `{:node <id> :node-type <τ> :T <0..1> :tension <Σ over incident edges> :edge-families […]}`.

### Relation types / identity

- Entry `:id` is stratum-namespaced (`s1/tension`, `s2/refactor-node`…). `:credit`/`:precision` keyed by `:id` — generalising `intrinsic-values`' class-keying (a mechanical migration, not a rewrite).
- `candidate-action.:type` references an `s2` entry `:id`. **If it references a class that doesn't exist → that absence is the niche-construction trigger (see below).**

### The ∇T reader, generalised (the car-1 design risk — resolved)

Today `mission_delta_t` is `{sorry → related-missions, T = mission phase}` — sorry-only, phase-only. Generalise to a **`tension-reader`** parameterised by: (a) a set of **target node-types** (not just sorries); (b) a configurable **incident-edge-family** set (partly there already: `related-mission`, `mission-cross-ref`, `file→mission`); (c) a **T-source per node-type** (mission→phase-table today; pluggable). It returns `tension-reading`s; the tension-proposer emits a `candidate-action` at high-tension nodes.

**"What action does a tension propose?" — resolved as a mapping `κ` that is itself a registry.** `κ` maps node-type τ → the action-class to propose (over today's S2: `open-mission`/`address-sorry`/`fire-pattern`). `κ` is data, so "tension at node-type X proposes action Y" is the same `register!` move. **The payoff:** when `κ(τ)` has *no existing S2 class* to propose into, that absence is exactly the **candidate-class signal** fed to `C-candidate-gen` — **Gap-1's false-floor becomes Gap-2's niche-construction trigger**, concretely and on-spine. (This is the `aif/niche-construction` "false-floor as signal" reading, now mechanised.)

### PSR

- **Pattern chosen:** composite — `aif/niche-construction` (warrant) over `exotic/full-lift-registry` (homoiconic entry + provenance) + `aif/candidate-pattern-action-space` (bounded set + inclusion reasons) + `aif/evidence-precision-registry` (precision axis) + `futon2.aif.intrinsic-values` (Beta credit).
- **Candidates considered:** bespoke per-gap features (rejected in ARGUE — reintroduces the false floor); single opaque-fn registry (rejected — full-lift-registry's homoiconicity is needed for recursion-in-data).
- **Rationale:** the four compose into one `registry-entry` shape serving S1 and S2 identically; homoiconicity is load-bearing for meta-meta-learning.
- **Confidence:** medium-high. The ∇T-reader generalisation + `κ`-as-registry is the freshest part — spike it in VERIFY.
