## 1. Findings banked (claude-10, 2026-06-27 — verified, don't re-derive)

### 1a. Provenance: the *want* side is populated; the *paired arrow* is the empty join

- **`meme.db` (the `meme.arrow` store, futon3a) is the right SHAPE but under-populated.** `arrows` table = `source_id → target_id` (a typed have→want arrow), `mode ∈ {construction, composition, analogy, untyped}`, `status ∈ {open, correlated, constructed}` (the BHK/sorry lifecycle). **But it holds only 17 arrows / 24 entities / 0 proposals / 0 promotions** — a seed/demo, *not* a mission-scale corpus.
- **M-goals-and-holes mined the C-vector (the WANT side), not paired arrows.** Per `goals-holes-readiness.html`: **263 derived C-entries** (stated 139 · incompleteness 81 · mess 43), each `{flavour, outcome-ref, preferred, weight, provenance, discharged-by, witness}`, in substrate-2 shape, **sim-only (0 :7071 writes)**. Producers: `futon6/scripts/c_vector.bb` (CPU, 3 channels) + `c_mine_joint.py` (應-voice GPU, stub-validated). A C-entry's `discharged-by` (the method that achieves the goal) points at **substrate-2's empty PROOF layer.**
- **The paired (have→want) arrow = the forward↔backward JOIN = substrate-2's empty PROOF layer.** Not populated. Owned by [[M-populate-substrate-2]] (D4) + [[M-operational-vocabulary]]'s discharge side. This is *why* `meme.db` is empty of mission arrows — the pairing was never built.
- **The only ACTUALLY-populated (have,want) pairs today: `futon6/data/diffsub-moves.edn` — 55 moves**, each `:have "X/derive" :want "X/argue"` + `:move/class :close-hole` + `:delta-g`/`:score`/`:prior`. Pattern-PHASE granularity (transitions *within* a pattern ns), generated by `diffsub_emit.py` (JAX). Already consumed by the rollout ΔG (`cascade_lane/rollout-g-for` filters on `:have`/`:want`).
- **Mission-level reconstruction material exists:** `futon6/data/mission-scope-trees/*.json` (e.g. `M-goals-and-holes.json`) — `scope-count-by-binder-type` + `scope-hyperedges` (the Skolem scope structure). Not have/want directly; reconstructable.

**Net:** Joe's hypothesis is right *by design* (the `arrows` table is exactly a have→want store) and unverified-*in-fact* for a real reason — the paired corpus is the un-built PROOF join. We have **wants** (C-vector / 293 `:sorry` / star-map), **haves** (methods / diffsub), and **55 phase-grain pairs**; we do **not** have mission-level paired arrows.

### 1b. The magnet works; cascade richness is gated by LIBRARY COVERAGE, not the mechanism

- Cascade-on-demand is real: `construct_cascade(psi)` (futon3a) embeds psi (minilm) as the magnet, then phylogeny-greedy coverage-saturation over the pattern pool. **No stored cascade corpus needed.**
- `cascade_lane.clj` currently feeds it only the **id-stem** (`mission->psi` strips `M-`/hyphens) — a *weak* magnet. A **meme-style** psi reliably pulls more on-topic, higher-wholeness patterns. Sweep (same budget 20, magnet = id-stem vs meme phrasing):

  | mission | id-stem | meme |
  |---|---|---|
  | M-value-creation-loop | size 1, F=−0.19 | size 3, **F=+0.90** ✅ |
  | M-symbol-grounding | size 2, F=−0.72 (off-topic iching) | size 1, F=−0.05 (thin) |
  | M-self-documenting-stack | size 1, F=−0.46 | size 2, F=−0.44 (on-topic, wholeness 0.46→1.16) |
  | M-live-geometric-stack | size 1, F=−0.48 (off-topic) | size 1, F=−0.16 |

- **The binding constraint is library coverage of the mission's territory.** Only 1/4 crossed F>0; the others got better-aimed but stayed thin because the patterns aren't there. So "enough cascades" = "enough *where the library covers the territory*" — a measurable coverage question, not yes/no.
