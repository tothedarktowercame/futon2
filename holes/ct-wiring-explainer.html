What a wiring diagram is — the DarkTower category-theory model

The fold turns a cascade into a wiring diagram (R16, the close-the-loop arc — see aif-wiring-explainer.html). But what is a wiring diagram, precisely? This is the semantics layer: DarkTower (real Lean in ~/code/mathlib4/DarkTower) answers it categorically. The payoff, in DarkTower's own words: one small interface — typed holes + filling + combs, sequenced by BV — types a mission, a paper, AND a fold's wiring as the same structure, not three theories. A wiring diagram is a comb of typed holes you fill and discharge under a coverage.

Click a primitive (gold) or an instance (blue) to see its real Lean signature, plain-English meaning, and status. Below: the two worked examples that are already 0-sorry in Lean, and the CLean bridge that renders EDN outlines into these types.

The three worked examples — one interface, three domains

① A mission 0-sorry ✓

DarkTower/Examples.lean §MissionExample · futon3c E-mission-head.md

The 8-phase lifecycle is a BV sequence (non-commutative spine); the scope and organism readings are one object held together by BV.copar; an unwritten phase is a typed hole hungry to be written.

def lifecycle : BV Phase :=
  BV.seq (head) (… (seq instantiate document))
def readings : BV Reading :=
  BV.copar (scope) (organism)
def ghostPhaseHole : TypedHole where
  poly := { A := Phase, B := fun _ => GhostDirection }
  satiety := fun p => if p = document
              then .payoff else .parse

document is graded payoff (hungry — the goal), the rest parse. Reassociating the spine is a checked BV.Step.

② A paper 0-sorry ✓

DarkTower/Examples.lean §PaperExample · arXiv 0809.2517 "Let H be a Hopf algebra"

An ungrounded symbol (H) is a typed hole hungry for a concept; a binder store grounds it; the grounding is the ungroundedBinder discharge polarity — and the answer is computed, not asserted.

def symbolHole : TypedHole where
  poly := { A := Symbol, B := fun _ => Concept }
  satiety := fun _ => .role
def groundingPolarity : DischargeKind :=
  .ungroundedBinder
-- answer by computation (rfl):
example : answers binderQuery binderStore
        = [expectedBinding] := rfl   -- ?concept = HopfAlgebra

Symbol grounding is a typed-hole discharge — the third member of DarkTower's discharge duality (sorry/proof · query/answer · symbol/binder).

③ A fold's wiring to build

the R16 fold (E-close-the-loop) · "a useful third example for a later file" — Examples.lean

Examples.lean explicitly names our fold — M-first-flights cascade → temporary-gap → wiring — as the intended third instance. A fold's wiring is a comb of typed holes: boxes consume/produce, the fold fills them, :policy-holes are the un-discharged directions, and a closing ΔG is a discharge. Same interface as ① and ②.

This is the bridge back to R16: the futon2.aif.fold contract ({:wiring :delta-g :policy-holes}) is the operational shadow of TypedHole/Fill/Discharge. Making the corpus DarkTower-shaped makes it Lean-checkable (0-sorry iff well-formed).

The CLean bridge — EDN outlines → these Lean types, 0-sorry-gated

CLean = the EDN image of a CT treatment

futon6/holes/excursions/E-clean.md · DarkTower/CLeanProofs.lean · clean_to_lean.py

CLean is "Clojure/EDN as Lean-sketch" — a comb of typed Poly interfaces with typed holes, without the proof-term weight. clean_to_lean.py renders *.clean.edn into the same BV/TypedHole/DischargeKind machinery; the render builds 0-sorry iff the CLean is well-formed — that is the correctness gate.

Each CLean carries a dual reading (:clean/copar): the informal pattern-tagged method spine and the formal comb — well-formedness is their coherence (every method tag is exactly one box, in order).

Why CT and not text embeddings (EXP-3)

E-clean.md · the de-scope finding

Text retrieval provably plateaus on structural similarity — EXP-3 found that the load-bearing signal for "are these two proofs/missions alike?" is the compositional shape (combs + typed holes), not the prose. No embedding-model size closes that gap.

This is exactly why the embedding fold (impl #3) is an ansatz, not the foundation: the structure is the real object; an embedding is a bet about computing over it efficiently. And it clarifies the division of labour with Rob's pipeline (Lean→neo4j+pgvector, graph+embedding indexing): our job is to emit structure-bearing artifacts (CLean / DarkTower-shaped wirings) his pipeline ingests — not to be the retriever.

Worked CLean examples — pulled back from the Linode math.CT run (102 papers)

102 math.CT arXiv papers, run on Linode, each turned into a CLean — a method-tagged spine + a comb of typed boxes + the dual copar reading, with the method/macro types assigned by LLaMA (mechanical otherwise), rendered to the DarkTower Lean types. The shape recurs hard: 98/102 share the construct-exploit-discharge macro (the rest: 3 contradiction-reduce, 1 induct-tower), and the spine draws from a ~10-tag closed vocabulary (reduce-to-known-result 170 · construct-auxiliary-object 163 · transport-along-symmetry 69 · compute-invariant 20 · local-to-global 13 · …). This is EXP-3's thesis made concrete: the compositional shape is real, recurring, and small — exactly the structure that embeddings (or Rob's pgvector index) should key on, not the prose. 1410.3218 — Birkhoff subcategories

categorical algebra · macro: construct-exploit-discharge
:clean/seq [:construct-auxiliary-object :reduce-to-known-result
            :local-to-global :transport-along-symmetry
            :reduce-to-known-result]
:clean/boxes
[{:id :e-birkhoff-subcategory :method :construct-auxiliary-object
  :text "Bc is a Birkhoff subcategory of Ac"
  :consumes [:k :d-circ-k-zero :d-factorises]   ; typed holes
  :produces :birkhoff-subcategory}]             ; filled position
:clean/wires
[{:from :e-birkhoff-subcategory :to :e-k-v-zero-closed
  :carries :birkhoff-subcategory}]              ; a Comb (wire)

Each box is a TypedHole — :consumes = its directions/holes, :produces = the filled position; each wire is a Comb; the spine is the BV.seq. The whole paper proof is one wiring diagram.
